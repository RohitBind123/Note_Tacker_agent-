# CentralAgent — API Reference

Base URL (prod): `https://centralagent-production-457c.up.railway.app`
Interactive docs (auto-generated by FastAPI): **`/docs`** (Swagger) and **`/redoc`**.

The happy path is **fully automatic** (calendar → poller → scheduler → email).
These endpoints are for health checks, manual control, and debugging.

Conventions: all timestamps are UTC ISO-8601. `native_meeting_id` is the
`abc-defg-hij` code from a Meet URL.

---

## Health

### `GET /health`
Liveness — process is up, no external calls.
```json
{ "status": "ok", "service": "centralagent", "version": "0.1.0", "env": "production" }
```

### `GET /healthz/db`
Readiness — verifies Postgres is reachable.
```json
{ "status": "ok", "database": "reachable" }
```

## Meetings (`/meetings`)

### `POST /meetings/dispatch`
Manually send a bot to a Meet URL (creates a new meeting row, status `joining`).
Body (`DispatchRequest`):
```json
{ "meet_url": "https://meet.google.com/abc-defg-hij",
  "title": "Optional", "organizer_email": "who@gets.email", "bot_name": "CentralAgent Notetaker" }
```
Returns `201` + `MeetingOut`. `422` if the Meet URL is invalid.
```bash
curl -X POST $BASE/meetings/dispatch -H 'Content-Type: application/json' \
  -d '{"meet_url":"https://meet.google.com/abc-defg-hij","organizer_email":"you@example.com"}'
```

### `GET /meetings?limit=&offset=`
List meetings (newest first). `limit` 1–100 (default 50). Returns `[MeetingOut]`.

### `GET /meetings/{id}`
One meeting → `MeetingOut`. `404` if not found.

### `POST /meetings/{id}/refresh`
Poll the provider once and reconcile our status (joining→active, ended→processing).
Returns `MeetingOut`.

### `POST /meetings/{id}/transcript`
Fetch the transcript from Vexa **and store it**. Returns `TranscriptOut`
(`segment_count`, `full_text`, `source`, `fetched_at`).

### `GET /meetings/{id}/transcript`
Return the stored transcript → `TranscriptOut`. `404` if none stored yet.

### `POST /meetings/{id}/analyze`
Run Gemini on the stored transcript and store the report → `ReportOut`
(`summary`, `decisions`, `action_items`, `risks`, `next_steps`, `model_used`).
Requires a stored transcript first.

### `GET /meetings/{id}/report`
Return the stored report → `ReportOut`. `404` `"report not generated yet"`.

### `POST /meetings/{id}/send-email`
Email the insights to the organizer and mark the meeting `completed`.
Returns `EmailResult`:
```json
{ "meeting_id": 103, "message_id": "19eb...", "status": "completed" }
```

### `POST /meetings/{id}/stop`
Stop the bot **and** move the meeting to `processing` immediately (the scheduler
then finalizes → transcript → Gemini → email on its next tick). Returns
`StopResult` `{ "meeting_id": 103, "stopped": true }`.

> The full manual pipeline, if you ever need to drive it by hand:
> `POST /{id}/transcript` → `POST /{id}/analyze` → `POST /{id}/send-email`.

## Admin / debug (`/admin`)

Internal — used to exercise the background flows without waiting for the loop.

### `POST /admin/poll-calendar`
Run one calendar poll now (detect + auto-RSVP + upsert). → `{ "upserted": N }`

### `POST /admin/scheduler-tick`
Run one full scheduler tick now (recover → dispatch → advance → process). →
`{ "status": "ticked" }`

## Webhooks (`/webhooks`)

### `POST /webhooks/google/calendar`
Google Calendar push receiver (`events.watch`). Header-only POST from Google;
validates `X-Goog-Channel-Token` against `CALENDAR_WEBHOOK_TOKEN`, then triggers a
calendar sync on a real change (`X-Goog-Resource-State != "sync"`). Always returns
`200` (so Google doesn't retry-storm). **Off in dev** (`CALENDAR_PUSH_ENABLED=false`);
see [CALENDAR_PUSH.md](./CALENDAR_PUSH.md).

## Schemas (shapes)

```text
MeetingOut    { id, google_event_id?, platform, native_meeting_id, meet_url,
                title?, organizer_email?, status, vexa_bot_id?, start_time?,
                end_time?, created_at, updated_at }
TranscriptOut { meeting_id, segment_count, full_text, source, fetched_at? }
ReportOut     { meeting_id, summary?, decisions?, action_items?, risks?,
                next_steps?, model_used?, created_at }
StopResult    { meeting_id, stopped }
EmailResult   { meeting_id, message_id, status }
```

`status` (MeetingStatus): `pending · scheduled · joining · active · processing ·
completed · cancelled · failed_join · failed_recording · failed_analysis ·
email_failed`. See the state machine in [ARCHITECTURE.md](./ARCHITECTURE.md#5-meeting-state-machine).
